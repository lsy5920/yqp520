# 云栖派官方网站

## 项目介绍
这是一个为“云栖派”打造的官方官网项目，用来展示门派世界观、立派全典、门规禁律、宗门日常与合并后的入派考核页，并提供带首页二维码的海报分享能力、问心考核结果海报能力、江湖名帖工作台能力、`云栖名册` 线上入册能力、公开名录固定招募海报能力与全局背景音乐播放能力。

当前首版已经落地以下核心内容：

- 公开前台已经包含 `首页`、`立派全典`、`门规与禁律`、`宗门日常`、`入派考核（内含原入派说明与问心考核）`、`云栖海报`、`江湖名帖`、`云栖名册登记`、`云栖名册公开名录` 与 `云栖名帖公开详情`
- `江湖名帖`独立创作页，支持 `8` 项核心字段填写、自动存草稿、文字版复制与按 `57 × 84` 比例输出的高清成图导出；新版已经改成沉浸式单屏工作台与单套“江湖正帖”视觉，去掉旧双模板和旧归档区，明显压缩成图空白，并且成图自带直达名帖页的二维码
- `云栖名册` 独立系统，支持 `公开登记入册 + Supabase 落库 + 执事审核 + 公开名录 + 云栖名帖分享`；现在已经统一成单一 `道号` 口径，并与问心考核绑定，只有最近一次考核合格后才能递交入册文牒，审核通过后自动转为正式入册名帖并进入公开名录
- `云栖名册公开名录` 页顶部保留固定版官方招募海报，采用 `1080 × 1350` 竖版成图、江湖门籍风格与二维码引导，可直接保存海报或原生分享；公开名录正文改成木签江湖墙，登记入册的道名会刻在一枚枚随机漂浮的高级木签上，点击道名会弹出公开信息小窗
- 公开名录、公开详情页与分享名帖始终遵循“最克制公开信息”策略，不展示 `俗家姓名 / 现居洞府 / 生年 / 俗务 / 微信号 / 其他社交号 / 闲暇时辰` 等敏感字段
- `Supabase Auth` 执事审核台，支持邮箱密码登录、四栏状态分组、按 `道号 / 文牒号` 快速检索、堂口筛选、详情抽屉查看全部原始字段，并支持全字段编辑、后台设置门中分工、后台自定义文牒号、彻底删除档案；抽屉打开后浏览器返回会优先退出编辑态，再收起详情，不会直接把人带离审核台
- 青金云海风格视觉、江湖门派气质、滚动动效、移动端适配
- 全局背景音乐系统，支持预加载检测、全屏江湖风格启播浮窗、异常停止后的再次引导、手动暂停 `15` 分钟后的重新检测，以及按时间点浮现的单句逐字漂浮歌词动效
- 右下角悬浮播放器，全端都支持手动收起与自动贴边；展开后会在 `2` 秒空闲后自动缩成贴边音乐图标，点击图标可再次展开
- 入派考核页已经和入门说明合并为单一入口；页面上半区只保留精简版入门要点，下半区直接进入固定 `30` 题、`7` 章、`10` 分钟的问心考核，支持自动交卷、章节直达、整卷题号速切、临时退出后继续作答与结果海报导出
- 问心考核只在浏览器本地记录，支持恢复未完成草稿，并且只保留最近一次正式交卷结果
- 交卷后自动生成结果页，展示总分、是否合格、单选与多选正确数、章节答题情况，并统一引导前往《立派全典》自行查阅原文
- 问心考核结果海报已接入，支持实时预览、保存图片、原生分享优先、失败自动回退下载，并默认带入派考核页中的问心考核区二维码
- 官方海报生成器，支持填写“同门称呼”和“一句寄语”，默认自动嵌入云栖首页二维码
- 云栖江湖名帖工作台，支持填写江湖名号、旧名或本名、来处、身份一句、江湖短签、入门缘起、留名一句与人像，自动生成更正式的竖向江湖正帖，成图自带扫码改帖二维码，并保留文字版复制与本机草稿恢复
- 海报门帖版式已再次重做，减少大面积空白，强化卷轴主视觉、门旨副区、山门题记、落款与二维码收束
- 海报中的二维码链接区已改为独立网址栏，长链接会自动换行完整展示，不再挤在二维码旁边截断
- 海报支持实时预览、保存图片、原生分享优先、失败自动回退下载
- 海报预览与最终下载内容保持一致，避免“页面看见一版、下载出来另一版”

## 安装部署教程
下面按“小白从零开始”的方式来写，建议严格按顺序操作。

### 第一步：进入项目目录
打开 PowerShell，输入下面的命令：

```powershell
cd "C:\Users\lanshiy\Documents\小亦伟大工程\yqp520"
```

### 第二步：安装依赖
第一次运行前，先安装项目依赖：

```powershell
npm install
```

如果这一步没有报错，说明项目所需依赖已经安装完成。

### 第三步：配置 Supabase 环境变量
云栖名册功能依赖 Supabase。先复制一份环境变量模板：

```powershell
Copy-Item .env.example .env.local
```

然后打开项目根目录下的 `.env.local`，填入你自己的 Supabase 配置：

```text
VITE_SUPABASE_URL=你的 Supabase 地址
VITE_SUPABASE_ANON_KEY=你的 Supabase 匿名键
```

如果你暂时不想启用云栖名册，也可以先不配；只是 `/roster`、`/roster/list`、`/roster/entry/:slug` 与审核台页面会直接提示你还没连上数据库。

如果你准备部署到 GitHub Pages，还需要在 GitHub 仓库的 `Settings -> Secrets and variables -> Actions -> Variables` 里额外创建这两个同名变量：

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

当前项目的 Pages 工作流已经改成构建前自动校验这两个仓库变量；如果没配，工作流会直接报错并提示补齐，避免线上站点打包成功却拿不到 Supabase 配置。

### 第四步：在 Supabase 执行名册 SQL
打开 Supabase 后台的 `SQL Editor`，把下面这个文件里的内容整段复制进去执行一次：

```text
supabase/yunqi_roster.sql
```

这一步会自动创建或迁移：

- `yunqi_roster_entries` 主表
- `yunqi_roster_admin_profiles` 执事资料表
- `yunqi_roster_review_logs` 审核日志表
- 公开登记、公开名录、公开详情与道号校验 RPC
- 后台全字段保存、删档、默认文牒号推荐与 RLS 权限规则

如果你之前已经跑过旧版名册 SQL，也要重新执行这份新版 SQL，它会把旧的 `江湖名号 + 法号` 字段迁移成统一的 `道号` 字段，并停用旧的自动牒号序列逻辑。

如果后台出现“编辑时道号没自动带出”“点击保存像没反应”这类现象，基本也说明这里还没有执行到最新版。因为旧库里没有 `daohao` 字段，也没有新版后台保存 RPC。

如果 SQL Editor 报 `cannot change return type of existing function`，说明你库里已经有旧版同名函数。现在项目里的新版 `supabase/yunqi_roster.sql` 已经补了“先删除旧函数，再重建”的兼容处理，使用最新本地脚本整段重新执行即可。

执行完 SQL 后，再去 Supabase 的 `Authentication` 里先创建执事账号，并按 SQL 文件末尾示例把该账号写入 `yunqi_roster_admin_profiles`，这样站内审核台才能登录使用。

### 第五步：启动本地开发环境
运行下面的命令：

```powershell
npm run dev
```

启动成功后，终端里会出现一个本地访问地址，通常类似：

```text
http://localhost:5173
```

把这个地址复制到浏览器里，就能看到官网页面。

### 第六步：打包生产版本
如果你要生成可部署的静态文件，运行：

```powershell
npm run build
```

打包成功后会生成 `dist` 文件夹，这里面就是可以部署到服务器上的静态文件。

### 第七步：本地预览打包结果
如果你想先看打包后的效果，再决定是否部署，可以运行：

```powershell
npm run preview
```

## GitHub 部署教程
如果你已经把项目推送到了 GitHub 仓库，可以直接按下面步骤部署到 GitHub Pages。

### 当前仓库信息
- GitHub 仓库：`https://github.com/lsy5920/yqp520`
- 预期访问地址：`https://lsy5920.github.io/yqp520/`

### 第一步：把最新代码推送到 `main` 分支
当前自动部署工作流监听的是 `main` 分支，所以先确认最新代码已经推上去：

```powershell
git add .
git commit -m "配置 GitHub Pages 部署"
git push origin main
```

如果你前面已经提交过，这里直接执行最后一条 `git push origin main` 也可以。

### 第二步：在 GitHub 仓库里开启 Pages
进入仓库网页后，按下面顺序操作：

1. 点击仓库顶部的 `Settings`
2. 点击左侧的 `Pages`
3. 找到 `Build and deployment`
4. 把来源切换成 `GitHub Actions`

这一步很关键，不切到 `GitHub Actions`，仓库里的自动部署工作流不会正式接管发布。

### 第三步：等待自动部署完成
项目里已经新增了自动部署文件：

`/.github/workflows/deploy-pages.yml`

当你推送到 `main` 分支后，GitHub 会自动完成：

1. 安装依赖
2. 执行 `npm run build:pages`
3. 生成 `dist` 产物
4. 自动发布到 GitHub Pages

你可以在仓库顶部的 `Actions` 页面里查看运行进度。当前工作流已经把发布任务改成排队等待，不会再因为连续推送而中途取消上一轮 Pages 发布；同时每个构建和发布任务都有 `10` 分钟超时兜底，避免一直卡住看不出结果。

### 第四步：打开站点
工作流成功后，访问下面这个地址即可：

`https://lsy5920.github.io/yqp520/`

如果第一次打开还没刷新出来，等几十秒再刷新一次通常就好了。

### 第五步：为什么这次配置能兼容 GitHub Pages
这次部署一共做了三层处理：

- Vite 构建时会自动使用仓库子路径 `/yqp520/`
- Vue Router 会读取这个基础路径，避免资源路径和路由路径错位
- 打包后会自动生成 `dist/404.html`，解决 GitHub Pages 直接刷新内页时的单页应用兜底问题

### 第六步：如果以后仓库名改了怎么办
如果你的仓库名以后不是 `yqp520` 了，要同步改下面这个文件：

`/.github/workflows/deploy-pages.yml`

把里面的：

```yaml
VITE_BASE_PATH: /yqp520/
```

改成新的仓库路径。比如仓库名改成 `yunqi-site`，这里就要改成：

```yaml
VITE_BASE_PATH: /yunqi-site/
```

不改的话，页面虽然能部署成功，但静态资源路径会错，页面会出现样式丢失或脚本 404。

## 使用教程

### 场景一：正常浏览官网
1. 启动项目后，默认会打开首页。
2. 顶部导航现在可以直接切到 `云栖名册`，页脚也同步补了这一入口。
3. 桌面端导航已经做过重新布局，右侧不会再留出一大片无意义空白。
4. 页面右下角有全局音乐播放器和返回顶部按钮。
5. 手机端页头会自动缩成折叠菜单，展开后的导航入口会改成两列卡片；页面里的多个普通操作按钮也会统一按两个一排显示，若刚好剩最后一个按钮，会自动独占整行，更适合单手点击和浏览。
6. 页面整体默认开启完整动效，不再提供导航栏“减少动效”开关。

### 场景二：使用背景音乐
1. 网站进入后会先自动预加载《云栖之缘》；一旦检测到曲目已经加载到可播放状态，就会弹出全屏江湖风格浮窗，提示“云栖之缘已备好”，轻点页面任意位置即可开始播放。
2. 如果背景音乐在已经加载好的情况下意外停下，页面会再次弹出全屏浮窗，引导你轻点任意位置继续播放。
3. 如果你是手动点击右下角播放器把音乐暂停，系统会记住暂停时刻和停下的位置；在 `15` 分钟后，如果音乐仍未继续，会重新检测并再次弹出引导浮窗。
4. 右下角播放器现在只保留播放、暂停、曲名和状态提示，不再提供“启播说明”和“静音”入口。
5. 播放《云栖之缘》时，页面会在对应时间点自动浮现歌词，当前只显示一句，并按“一个字一个字慢慢漂浮出来”的方式展示；手机端与桌面端现在都会贴近页面底部显示；当前首句大约会在播放到 `00:34` 时出现，所以前半分钟没有歌词属于正常现象。
6. 手机端和桌面端现在统一使用同一套播放器交互，都会先完整展开。
7. 如果你 `2` 秒内没有继续操作，它会自动缩成贴右侧边缘的音乐图标。
8. 点击“收起”按钮可以立即贴边缩回。
9. 点击这个音乐图标后，播放器会再次展开。
9. 系统会记住你的播放状态，刷新页面后会尽量恢复。

### 场景三：查看入派考核并参加问心考核
1. 进入 `入派考核` 页面；现在入门说明和问心考核已经合并在同一页里，旧的 `/assessment` 链接也会自动跳回这里的考核区。
2. 开考前可以先填写“同门称呼”，这项是可选的，只用于成绩展示和结果海报。
3. 点击“正式开考”后会以弹窗形式打开答题区，并进入 `10` 分钟倒计时。
4. 整套考核按 `7` 个章节分步作答，弹窗顶部会显示当前章节、总题进度、未答题数和剩余时间，并提供“临时退出”入口。
5. 手机端会把章节进度改成横向滑动条，同时取消高面板吸顶，避免顶部概览压住题目内容。
6. 弹窗会直接覆盖在页面上，不会再把整页拉到题目区；现在答题区改成“一次只看一题”的工作台，手机端会把弹窗做成固定高度面板，正文由弹窗自己滚动。
7. 顶部章节卡片现在可以直接点击切换，整卷 `30` 题的题号也可以直接点击跳转；题号颜色会区分“当前题 / 已作答 / 未作答”。
8. 手机端答题时，“当前章作答情况”和“题号速切”两块默认会先收起，只保留一行摘要；需要时点一下就能展开，不会一上来占掉太多高度。
9. 开考后页面会自动滚到当前题目区域；跨章节切换时也会稳定回到当前工作区，不会停在题目中段或把外层页面带着乱跳。
10. 如果临时有事，可以先点“临时退出”收起答题弹窗；倒计时会继续走，之后再点“继续作答”即可回到刚才离开的位置。
11. 手机端点按答题选项时，现在会自动同步释放触屏焦点，并关闭点按位移动画和滚动锚点，避免题目区域突然滑动一下。
12. 页面刷新后会优先恢复未完成草稿，倒计时不会重置。
13. 倒计时结束会自动交卷；如果你提前答完，也可以手动点击“交卷”。
14. 交卷后会自动进入结果页，显示总分、是否合格、答对题数与章节成绩；如需对照原文，统一点击“去看立派全典”自行查阅。
15. 手机端结果页现在会把“答题统计卡”和“章节成绩卡”优先排成双列；总分大牌、结果说明和主操作按钮这类主块仍保持单列，避免信息层级混乱。
16. 成绩只保存在浏览器本地，而且只保留最近一次正式交卷结果。

### 场景四：生成考核结果海报
1. 完成问心考核并交卷后，结果页下方会自动出现“结果海报”区域。
2. 结果海报会自动带上同门称呼、分数、是否合格、完成日期、用时和赴考二维码。
3. 点击“保存图片”会下载成绩海报。
4. 点击“分享或下载”时：
   - 如果当前设备支持系统原生分享，会优先拉起分享面板。
   - 如果当前设备不支持原生分享，会自动回退为下载图片。
5. 结果海报二维码默认会跳转到 `入派考核` 页里的问心考核区。

### 场景五：生成分享海报
1. 进入 `云栖海报` 页面。
2. 在“同门称呼”中输入你想显示的称呼。
3. 在“一句寄语”中输入你想写的话。
4. 右侧海报会实时预览。
5. 当前海报会默认带上“云栖首页二维码”，别人扫码后会直接回到山门首页。
6. 点击“保存图片”会下载海报。
7. 点击“分享或下载”时：
   - 如果当前设备支持系统原生分享，会优先拉起分享面板。
   - 如果当前设备不支持原生分享，会自动回退为下载图片。
8. 现在页面预览与最终下载已经统一为同一份海报内容源，看到什么，下载下来就是什么。
9. 手机端海报预览现在也会和电脑端保持同一套预览外观，不再单独改成另一组圆角和宽度规则。

### 场景六：海报输入规则说明
- 同门称呼建议不超过 `12` 个字
- 一句寄语建议不超过 `60` 个字
- 组件会自动过滤一部分容易干扰排版的特殊字符
- 如果你什么都不填，会自动使用默认兜底文案，不会直接报错
- 如果二维码还没生成好，系统会先提示等待，不会导出一张缺二维码的海报

### 场景七：生成江湖名帖
1. 先进入 `入派考核` 页面，或直接点首页里的“生成江湖帖”按钮。
2. 点击入口后，会进入独立的 `江湖名帖` 页。
3. 按顺序填写江湖名号、旧名或本名、来处、身份一句、江湖短签、入门缘起、留名一句，并可选上传人像。
4. 桌面端会以“左侧成品预览、右侧分组填写”的方式编辑，预览区会尽量铺满左侧面板并保持成品比例；手机端会先显示成品预览，再往下填写分组内容，手机与电脑会共用同一套成品排版逻辑。
5. 点击“生成江湖帖”后，系统会为当前草稿正式顺延帖号，并记录立帖时间。
6. 点击“保存成图”会导出一张按 `57 × 84` 比例输出的竖向江湖正帖图片，当前默认成图像素为 `1140 × 1680`，并且成图右下区域会带上直达 `江湖名帖` 页的二维码。
7. 点击“复制文字版”可以直接把简洁帖文发到微信群。
8. 草稿会自动保存在浏览器本地；刷新页面后会优先恢复当前草稿，但不会再展示旧式归档列表。

### 场景八：使用云栖名册
1. 如果你刚考核通过，结果页会出现主按钮“前往云栖名册登记”，点进去就能直接递交入册文牒。
2. 也可以从顶部导航直接进入 `云栖名册`；如果还没有通过问心考核，页面会先显示拦截引导，不会直接放出登记表单。
3. 只有最近一次问心考核合格后，登记页才会展开 `弟子名籍 / 门派司职 / 传讯方式 / 所长与愿 / 入派誓约 / 提交确认` 六段长表单；顶部只保留“真实信息提醒”，各输入项自己的提示文案仍会继续保留。
4. 登记页现在已经把 `江湖名号 + 法号` 统一成单一 `道号` 字段；道号支持自由拟定，不再强制以 `云` 或 `栖` 开头，也不再限制固定两字。
5. 当前名册登记里的所有输入框和文本框都按必填处理，并新增了 `性别` 必选项；如果有任意一项留空或未选，前端和 Supabase 接口都会直接拦下。
6. 登记页顶部新增了“真实信息提醒”区；`俗家姓名 / 生年 / 核心传讯` 都要求如实填写，提交前还必须勾选真实性确认。
7. `道号` 会先做基础长度校验，再调用 Supabase RPC 做重名检查；若已被占用，就不能继续递交。
8. 提交成功后会立刻跳到 `/roster/entry/:publicSlug`，先看到一张 `待审核` 名帖。
9. `待审核` 名帖详情页默认收起海报功能，点击“展开海报功能”后才显示预览、保存图片、复制链接和原生分享；`准予入册` 后会自动切成正式版名帖并进入公开名录。
10. `暂缓入册` 与 `不予收录` 的详情页仍可查看公开批语，但不再提供名帖导出。
11. `公开名录` 页只展示已经审核通过的条目，支持按 `道号 / 文牒号` 搜索，也支持按堂口筛选；名录正文以“木签江湖墙”展示，点击木签上的道名会弹出公开信息小窗，可查看本心、所长、雅事、状态、编号和执事批语。
12. `公开名录` 顶部现在还有一张固定版“云栖名册登记招募海报”，可直接保存海报或分享海报；二维码会先把人带到公开名录，再决定是否继续入册登记。
13. `审核台` 登录地址是 `/roster/admin/login`，只允许已经加入白名单的 Supabase 执事账号使用。
14. `审核台` 现在支持直接编辑所有已填信息、修改状态、后台设置“同门 / 宗主 / 云司·文司 / 云司·事司 / 云司·财司”等门中分工、后台自定义正式文牒号；如果你不手填文牒号，系统会自动带出“当前最大正式牒号 + 1”的建议值，移动端列表也统一改成优先显示道号，不再用堂口做前缀。
15. `审核台` 抽屉打开后，浏览器返回键、顶部“返回列表”和点击遮罩都会优先按“退一层”的方式处理：先退出编辑态，再收起详情，不会直接离开执事后台。
16. `审核台` 支持彻底删除任意一条档案；删除后，公开页和后台都会查不到这条记录，且不可恢复。
17. `审核台` 在长时间停留后再次保存时，会先主动检查并刷新登录状态；如果会话已经过期或刷新卡住，页面会尽快提示重新登录，不会一直停留在“保存中...”。
18. `公开名录` 页顶部和站点页脚都已补上“执事管理入口 / 执事审核”入口，桌面端会直接跳到登录页，手机端也能直接点击进入。
19. `性别` 与后台设置的 `门中分工` 会进入公开名录、公开详情页和云名帖；但俗家姓名、现居洞府、微信号等敏感信息依旧不会公开。

## 入派考核页面使用教程

### 入派考核页面包含哪些能力
当前首版已经把入门说明与问心考核收成同一页，落地下面这些能力：

- 固定 `30` 题标准题卷
- 按 `7` 个章节分步作答
- 单题作答工作台
- 章节点击直达
- 整卷题号状态速切
- 手机端章节信息与题号区默认收起
- `10` 分钟倒计时
- 超时自动交卷
- 本地草稿恢复
- 最近一次正式成绩本地记录
- 结果页直达《立派全典》查阅原文
- 考核结果海报预览、保存、分享
- 手机端结果统计与章节成绩双列排版

### 问心考核评分规则
- 单选题 `20` 题，每题 `3` 分，共 `60` 分
- 多选题 `10` 题，每题 `4` 分，共 `40` 分
- 多选题必须全对才得分，少选和错选都不得分
- 总分 `100` 分
- 合格线 `80` 分

### 问心考核答题流程
1. 进入 `入派考核` 页面并下滑到问心考核区，旧的 `/assessment` 链接也会自动落到这里
2. 可选填写“同门称呼”
3. 点击“正式开考”
4. 在弹窗里按章节逐步答题，并一次只看一题
5. 手机端默认会先收起“当前章作答情况”和“题号速切”，需要时再手动展开
6. 点击章节卡片或题号直接跳到指定题目
7. 根据题号颜色判断当前题、已答题和未答题
8. 手机端点击选项时不会再让题目卡突然滑动或跳一下
9. 手动交卷，或等倒计时结束自动交卷
10. 在结果页查看成绩摘要；如需对照原文，点击“去看立派全典”，合格后再前往云栖名册登记

### 本地记录规则
- 未交卷时会保存草稿到浏览器本地
- 刷新页面后会优先恢复未完成考试
- 倒计时会按照原来的截止时间继续走，不会重新开始
- 正式交卷后只保留最近一次成绩
- 重新考核时不会保留上一轮的答题草稿

### 补考规则说明
首版只会在未合格时显示“建议三日后补考”的提醒，不会在浏览器本地强行锁定三天。这样做是为了先保证流程稳定，不让本地时间或浏览器权限问题误伤正常使用。

### 结果海报说明
- 结果海报会自动带上 `入派考核` 页里的问心考核区二维码
- 海报内容固定包含同门称呼、分数、是否合格、完成时间、用时和官方结果文案
- 海报预览和下载使用同一份内容源，看到什么，下载下来就是什么

## 海报功能使用教程

### 海报支持哪些内容
当前首版只开放两项输入：

- `同门称呼`
- `一句寄语`

海报里的门派主视觉、主标题、箴言、二维码区域、底部落款都由系统统一控制，首版不开放自由换底图、拖拽和改字号。当前版式已经重做成更饱满的“三段式门帖”结构，上段立题、中段卷轴与门旨并列、下段用山门题记与二维码落款收束，整体留白明显少于旧版；网址也放进了独立栏位，长链接会自动换行完整显示。

### 海报保存流程
1. 输入称呼和寄语
2. 等右侧预览更新完成
3. 点击“保存图片”
4. 浏览器会开始下载 `png` 图片

### 海报分享流程
1. 点击“分享或下载”
2. 如果浏览器支持 `Web Share API` 且支持分享文件，会直接拉起系统分享
3. 如果浏览器不支持，会自动改为下载图片

### 海报导出一致性说明
为了避免预览和下载不一致，现在海报页面采用的是“单一内容源”方案：

- 页面中看到的预览，本质上是原尺寸海报的缩放展示
- 下载导出时，会从这同一份海报内容克隆导出
- 所以文字排版、二维码位置、留白和视觉结构都会保持一致
- 手机端预览壳层也和电脑端统一，避免同一张海报在不同设备上看起来像两套预览
- 当前版本也已修复导出海报底部被额外撑出大块空白的问题

## 背景音乐配置教程

### 音乐配置文件在哪里
背景音乐配置写在：

`src/data/siteContent.ts`

背景音乐的歌词时间轴也一起写在这个文件里的 `musicTracks` 配置中。

### 正式音频文件放哪里
正式音频文件建议放在：

`public/media/`

当前项目已经接入的正式音频文件是：

`public/media/云栖之缘.mp3`

### 当前音乐配置示例
```ts
musicTracks: [
  {
    id: 'yunqi-theme',
    name: '云栖之缘',
    filePath: 'media/云栖之缘.mp3',
    coverText: '山门清音 · 云栖之缘',
    enabled: true,
    defaultVolume: 0.56,
  },
]
```~

### 每个字段是什么意思
- `id`：曲目的唯一标识
- `name`：播放器里显示的曲目名
- `filePath`：音频文件路径，推荐写成 `media/文件名.mp3` 这种不带开头斜杠的形式，方便本地和 GitHub Pages 子路径部署共用
- `coverText`：播放器里的补充说明文案
- `enabled`：是否启用当前曲目
- `defaultVolume`：默认音量，建议写 `0` 到 `1` 之间的小数

如果旧配置里还保留了 `onboarding` 这类历史字段，也不会影响现在的播放器运行，只是当前版本已经不再读取它。

### 为什么 GitHub Pages 现在也能正常播音乐
这次已经把播放器内部的音频地址解析方式改成“自动跟随站点基础路径”：

- 本地开发时会按当前本地地址读取
- 部署到 GitHub Pages 时会自动跟着 `/yqp520/` 子路径读取
- 所以不需要你再手动写死完整仓库地址

### 为什么有时不会自动播放
浏览器通常会限制没有用户手势时的自动播放，所以新版改成先预加载、再弹出全屏浮窗引导你轻点页面任意位置播放；这样既能保证音乐已经备好，也能绕开大多数浏览器的自动播放限制。

## 常见问题排查

### 1. 运行 `npm install` 失败怎么办？
先确认本机已经安装 `Node.js 25.8.0` 和 `npm 11.11.0`。如果你装的是更老版本，建议先升级再安装。

### 2. 运行 `npm run dev` 后打不开页面怎么办？
先看终端有没有报错。还要注意端口不一定永远是 `5173`，如果被占用，Vite 会自动换端口，请以终端实际输出地址为准。

### 3. 为什么背景音乐没有马上响？
这通常不是项目坏了，而是浏览器拦截了自动播放，或者音乐在后台被意外打断。现在页面会在音乐备好或异常停止时自动弹出全屏引导浮窗，你只要轻点页面任意位置，通常就能恢复播放。

### 4. 为什么播放后歌词没有立刻出现？
这是当前歌词时间轴本身的正常表现。《云栖之缘》的首句歌词大约从播放到 `00:34` 才开始浮现，所以前面一小段只有音乐没有歌词，不代表功能失效。

### 5. 为什么播放器自动缩小了？
这是当前全端统一交互。播放器完整展开后，如果 `2` 秒内没有继续操作，会自动缩成右侧音乐图标，避免遮挡页面内容；你也可以直接点“收起”按钮手动贴边。点击图标即可重新展开。

### 6. 为什么我看到的海报和下载下来的海报以前不一样？
旧方案是两份不同节点分别负责预览和导出，容易出现版式差异。当前版本已经修复为单一内容源，所以现在看到什么就会下载什么。

### 7. 为什么点“分享或下载”时没有弹出系统分享，而是直接下载了图片？
这是因为当前浏览器或设备不支持文件级原生分享。项目已经做了自动兜底，会直接改为下载海报，这不算故障。

### 7. 二维码为什么没有马上出现？
二维码会在页面进入后自动生成。如果页面刚打开、网络环境较慢，或者浏览器状态不稳定，系统会先等待二维码就绪，再允许导出。

### 8. 我改了文案、音乐配置或样式，页面没变化怎么办？
如果你改的是源码文件，请刷新页面，必要时重启 `npm run dev`。如果你改的是部署文件，还要重新执行一次 `npm run build`。

### 9. 为什么刷新页面后考核还在继续，不会重新开始？
这是当前版本的正常行为。未交卷时，系统会把答题进度、开始时间和截止时间保存到浏览器本地，所以刷新后会优先恢复草稿，并按原来的倒计时继续。

### 10. 为什么我已经交卷了，再进考核页还是看到上次成绩？
这也是当前版本的正常行为。首版会把“最近一次正式交卷结果”保存在本地，方便你回来继续看成绩摘要、立派全典查阅入口和结果海报。点击“重新考核”后会开启新一轮答题。

### 11. 为什么结果海报扫码后会跳到入派考核页里的问心考核区？
因为结果海报本身就是为了传播问心考核入口，所以二维码默认落点固定为 `入派考核` 页里的考核区，而不是首页。

### 12. 为什么 GitHub Pages 打开后资源 404 或页面样式丢失？
最常见原因是仓库子路径不对。当前仓库名是 `yqp520`，所以 GitHub Pages 构建时必须使用：

`/yqp520/`

如果你以后改了仓库名，要同步修改 `/.github/workflows/deploy-pages.yml` 里的 `VITE_BASE_PATH`。

### 13. 为什么 GitHub Pages 直接刷新某个内页会报 404？
当前项目已经在构建时自动生成 `dist/404.html` 做单页应用兜底。只要工作流正常执行并且 GitHub Pages 来源设置成 `GitHub Actions`，这个问题一般不会再出现。如果仍然出现，优先检查 Pages 来源是不是还停留在旧的分支发布模式。

### 14. 为什么部署日志一直显示 `Current status: updating_pages`？
这通常是 GitHub Pages 还在更新页面，或者短时间内连续推送导致上一轮发布被打断。当前工作流已经改成 `cancel-in-progress: false`，后续发布会排队执行，不会再主动取消正在发布的任务；同时 `build` 和 `deploy` 都加了 `10` 分钟超时，超过时间会明确失败，方便重新运行。

如果你刚刚已经看到旧任务一直循环，可以先在 GitHub 的 `Actions` 页面取消那一次旧运行，然后重新点 `Run workflow`，或者再推送一次最新代码触发新部署。重新部署前请确认 `Settings -> Pages` 的来源仍然是 `GitHub Actions`。

### 15. 为什么江湖名帖刷新后没有恢复草稿？
先看页面提示里的存储状态。如果浏览器禁用了本机存储、处于无痕隐私模式，或者人像文件太大导致本机空间不足，系统可能会退回到当前会话模式。这种情况下，当前页面还能继续用，但刷新后草稿不一定能完整恢复。

### 16. 为什么头像上传后显示不正常或保存失败？
请尽量上传常见的图片格式，比如 `jpg`、`png`。如果图片特别大，先换一张更小的图再试；系统会自动压缩，但如果浏览器本机存储空间太紧张，还是可能保存失败。若只是想先生成名帖，也可以不上传头像，系统会自动使用占位图。

### 17. 为什么云栖名册页面提示没有配置 Supabase？
如果是本地开发环境，这通常说明你还没在项目根目录创建 `.env.local`，或者没有把 `VITE_SUPABASE_URL` 和 `VITE_SUPABASE_ANON_KEY` 填进去。先复制 `.env.example`，再补齐这两个变量，重启 `npm run dev` 后再打开 `/roster` 相关页面。

如果是 GitHub Pages 线上站点看到这条提示，则多半是仓库的 `Actions Variables` 里还没有配置这两个同名变量，或者配置后还没重新触发部署。当前工作流已经加了部署前校验，补齐变量后重新运行部署即可。

### 18. 为什么执事账号能登录 Supabase Auth，却进不了审核台？
先确认这个账号已经在 Supabase `Authentication` 里创建成功；再检查 `yunqi_roster_admin_profiles` 表里有没有对应的 `user_id`、`email`、`display_name`，并且 `is_active` 是否为 `true`。站内审核台不会只认邮箱密码，还会额外校验这张白名单资料表。

### 19. 为什么手机端执事登录会短暂显示“登录中”，随后又回到未登录？
这通常是移动端浏览器在刚登录成功时，会话和白名单资料同步有一点延迟。当前代码已经改成优先使用登录接口直接返回的会话，并补了一层管理员资料重试读取逻辑。若线上仍出现旧现象，请先确认你已经把最新代码重新部署到 GitHub Pages，再重新尝试登录。

### 20. 为什么后台编辑时没有自动带出道号，或者点击保存像没反应？
这通常说明当前 Supabase 还停留在旧版名册结构。旧表里还是 `jianghu_name / requested_style_name / effective_style_name` 这一套字段，没有新版统一口径的 `daohao` 列，也没有 `admin_save_roster_entry`、`admin_delete_roster_entry`、`get_next_roster_entry_no` 这些新版后台 RPC。

当前前端已经补了旧字段回填兼容，并且会在抽屉里直接提示是否命中了旧库问题；但要真正恢复后台保存、删档和新版检索能力，仍然需要回到 Supabase 的 `SQL Editor`，重新执行项目里的 `supabase/yunqi_roster.sql`。执行完成后刷新后台页面，再进入编辑模式即可看到统一后的道号与正常的保存反馈。

### 20. 为什么在 Supabase 执行名册 SQL 时会提示 `cannot change return type of existing function`？
这是因为数据库里已经存在旧版同名函数，而 PostgreSQL 不允许直接用 `create or replace function` 修改 `returns table` 的返回结构。现在项目里的最新 [supabase/yunqi_roster.sql](C:/Users/lanshiy/Documents/小亦伟大工程/yqp520/supabase/yunqi_roster.sql) 已经补了兼容处理，会先删除这些旧函数，再重建新版函数。

如果你此刻正卡在 SQL Editor，不想等整份脚本替换，也可以先手工执行下面这些删除语句，再重新跑整份脚本：

```sql
drop function if exists public.check_roster_daohao_available(text);
drop function if exists public.submit_roster_entry(jsonb);
drop function if exists public.list_public_roster_entries(text, text, integer, integer);
drop function if exists public.get_public_roster_entry_by_slug(text);
drop function if exists public.get_next_roster_entry_no();
drop function if exists public.admin_save_roster_entry(jsonb);
drop function if exists public.admin_delete_roster_entry(uuid);
```

### 21. 为什么执事后台停留很久后，点击保存会一直显示“保存中...”？
这通常和登录会话过期、浏览器后台休眠太久、或者网络请求卡住有关。当前代码已经补上两层兜底：一层是在保存、删档、拉日志、取建议文牒号前，先主动检查并刷新执事登录状态；另一层是给后台请求增加超时保护，避免按钮一直被锁住。

如果你在线上站点仍然看到旧现象，先确认已经重新部署最新代码；若已经是最新版本，按提示刷新页面重新登录审核台后，再继续保存即可。
